# network-ai — Design Context

This file is read by every design-related Claude session in this repo. It is
the source of truth for who network-ai is for, how it should feel, and what
it should never become. Update it when the answers to these questions change.

## Design Context

### Users

network-ai is a personal-network mapper. The primary user is a builder /
operator who wants a calm, fast, considered surface for everything they
remember about the people in their network — contacts, their warmth (1–5),
city, tags, free-form notes, and the assets each person (or "we") can offer.

The user interacts through a chat agent on the left and a reviewable
accordion of contacts + assets on the right. The agent does the writing;
the right pane is the durable record.

Sessions are short and frequent: drop a quick observation about someone,
scan a few contacts, ask "who do I know that could help with X." The
interface should reward returning daily.

### Brand personality

**Three words**: _considered, calm, fast._

- **Considered**: every line of typography earns its place; nothing decorative;
  prose-friendly notes display.
- **Calm**: low-contrast surfaces, generous whitespace, restrained color,
  motion that decelerates and disappears.
- **Fast**: optimistic updates, sub-100ms perceived latency on every
  interaction, no spinners where a skeleton or instant-state will do.

The emotional goal is the feeling of using a personal notebook that is
quietly aware of itself — never demanding attention, always ready.

### Aesthetic direction

**Reference**: Vercel.com product surfaces (not the marketing site), Linear
without the density, Are.na's editorial care.

- **Type**: Geist Sans body, Geist Mono _only_ for IDs, durations, and other
  inherently technical strings. Sizes set via a clamp-driven scale.
- **Color**: monochromatic. Off-white background (`#fafafa`), warm dark
  foreground (`#0a0a0a`), hairline borders (`oklch(96% 0 0)`), muted text
  (`oklch(48% 0 0)`). One accent for emphasis (deep emerald `#047857` —
  used for warmth-1 dots, active states, and AI-action confirmation tags).
- **Density**: generous (Vercel-like, not Linear-like). Contact rows are
  taller. Notes wrap with comfortable measure. The whole layout breathes.
- **Borders**: hairline 1px in the lightest neutral. Never harder than that.
  Cards rarely shadow; instead, lift via a subtle inset on hover.
- **Radii**: small (4px) on inputs + pills; medium (8px) on cards; never
  fully pill-shaped except for the warmth dots.
- **Iconography**: minimal. When used, monoline 1.5px stroke, lucide-react.
  No emoji as decoration. No fancy icon-above-heading layouts.

**Theme**: light only. (User decision — no dark mode planned.)

### Anti-references (NEVER ship)

- AI slop in any form: purple-to-blue gradients, glassmorphism, neon on
  dark, cyan-on-black, sparkle decorations, "AI-generated" patina.
- Card grids of equal-sized rounded rectangles.
- Cards nested in cards.
- Modals for anything that fits inline.
- Bouncy / elastic / spring-overshoot easings.
- Generic icon-above-heading templated layouts.
- Drop-shadow + gradient-background hero cards.
- Animated layout properties (height, width, padding). Transform + opacity only.
- Bright primary buttons for every action. Hierarchy via type weight + color
  contrast, not button color.

### Design principles

1. **Type carries the design.** Hierarchy, mood, and personality come from
   the Geist scale + weight contrast. Stop reaching for decoration; reach
   for a different weight.
2. **One pane never pushes the other.** The viewport is split 50/50 at the
   top, both panes own their own scroll, the page itself is fixed-height.
3. **Optimistic by default.** When the agent does something, the right pane
   reflects it before the server confirms. Rollback on failure.
4. **Every AI action is reviewable + revertable.** Tool calls render as
   structured cards in the chat with: what changed, in human language;
   a "Jump to" link that scrolls + highlights the right-pane row;
   for destructive calls, an Undo affordance.
5. **Motion explains.** Use it to convey what happened (a contact slid in
   from the right because the agent added it) — never as decoration. All
   motion is ≤250ms, ease-out, on transform + opacity only. Respect
   `prefers-reduced-motion`.

### Accessibility

- WCAG 2.2 AA contrast on text and interactive surfaces.
- Every interactive element has a visible focus ring (1.5px solid
  `oklch(60% 0.18 145)` with 2px offset).
- `prefers-reduced-motion: reduce` disables every non-essential animation;
  state changes become instant.
- Keyboard: `/` focuses composer; `Esc` closes any open inline editor;
  Tab order follows reading order.
- Composer textarea is labelled; tool-call cards have ARIA live regions so
  screen readers announce "added Anna Svensson", "deleted Bo Larsson", etc.

### What "done" looks like

A user opens the deployed site, signs in, types `"add Anna Svensson…"` and:

- The composer's text-area never grew the page; the chat scrolled internally.
- The assistant's bubble streamed in token-by-token with a quiet cursor.
- A small structured card appeared in the chat: _"Added Anna Svensson —
  warmth 1, Göteborg, 2 tags. [Jump to ↗]"_.
- Before the network round-trip completed, Anna's row appeared on the right
  with a soft fade + slide. The "Jump to" link, when clicked later, scrolls
  the right pane to her row and highlights it for 1.2s.
- Editing Anna's notes saves silently on blur with a 600ms debounce. No
  spinner. A tiny _"saved · 220ms ago"_ timestamp appears under the field.
- Deleting Anna replaces her row with a small "Undo" banner that auto-
  dismisses after 5s; the row gracefully animates out.

When that flow feels like one continuous, calm gesture — we're done.
